# CLAUDE.md — UI design rules for cleancut-legal

Read this file before designing or modifying any marketing UI in this repo.
The rules below exist to stop AI agents (including you) from shipping
designs that look like they were generated by an AI.

---

## The one-line test

> Would a designer paid $5,000 produce this exact pattern, **or** is this
> the obvious thing every AI generates when asked for a marketing page?
>
> If the second — change it.

---

## What "looks AI" actually means

Pattern-saturation. Symmetry-obsession. Every element optimized to the
lowest-risk choice. Safe, balanced, consistent to the point of being
unmemorable.

Human designers break pattern *intentionally* to create **one** moment
users remember 10 minutes after they leave.

A page with no memorable moment is an AI page, regardless of how polished
each element looks individually.

---

## When this file applies

| File | Apply these rules? |
| --- | --- |
| `index.html` (marketing) | **YES** |
| `privacy.html`, `terms.html`, `support.html` (legal) | **NO** — predictable is correct for legal pages. Single column. Readable. Boring. Do not add personality, do not break grid, do not stylize. |

## The 11 tells

Each rule has a project-grounded example. `DON'T` points at code already in
this repo that exhibits the tell. `DO` points at the human-designed
counter-example (often `aurora_redesigned_sections.html`, which is a
working reference of what "not AI" looks like).

### 1. Gradient-word-per-headline

**Rule:** At most ONE gradient-word on the whole page, applied to the
single most important phrase. Not one per section.

- **DON'T:** `index.html:83, 116, 251` — three uses of `.ai-gradient-text`.
  The gradient on "7:32" (phone screenshot) and "Apply Cuts" (CTA chip)
  dilute the gradient on "All on your iPhone." The trick stops being
  emphasis once it appears in three places.
- **DO:** `aurora_redesigned_sections.html:82-87` — the inline comment
  `/* Only ONE gradient word in the whole page */` makes the discipline
  explicit. Only the word "comfortable" gets the gradient.

### 2. Identical all-caps section eyebrows

**Rule:** Don't introduce every section the same way. Vary how each
section opens — eyebrow on some, bare headline on others, a badge or
quote on one.

- **DO (already correct in this repo):** Hero has an eyebrow
  ("A 30-min recording → 22-min edit"), but `<h2>Three tools, one phone.</h2>`,
  `<h2>Pricing</h2>`, `<h2>FAQ</h2>` open bare. That variation is good — keep it.
- **DON'T:** Don't "fix" the bare `h2`s by adding eyebrows everywhere.
  Don't make every section open with a small uppercase label.

### 3. Cloned feature-card grids

**Rule:** Feature grids must have hierarchy. One card bigger / different
shape / different color to signal the hero feature.

- **DON'T:** `index.html:261-291` — Smart Cut / Denoise / Merge are three
  identical cards (same size, same padding, same gradient strip).
  Smart Cut is the headline feature; it should be physically bigger.
- **DO:** `aurora_redesigned_sections.html:491-563` — BIG card
  (2-col × 2-row, hosts an interactive demo) + TALL card (1-col × 2-row,
  iOS integration list) + 2 small cards beneath.

### 4. Default pricing-card template

**Rule:** A single centered card OR a 4-tier symmetric grid is the AI
pricing default. If you ship it, justify it. Most designs should reconsider.

- **DON'T:** `index.html:322-379` + `style.css:459`
  (`grid-template-columns: repeat(4, 1fr)`) — Free / Monthly / Yearly
  highlighted / Lifetime in a perfect row. This is the most-templated
  pricing layout in AI's training data.
- **DO options:**
  - Two cards (Free + Pro), with Monthly/Yearly/Lifetime as inline toggles inside Pro
  - Three cards with Yearly visibly larger (off-grid)
  - Drop Monthly entirely

### 5. Default Lucide / Heroicons

**Rule:** Don't ship default icon-library SVGs as the product's visual
identity. Use geometric primitives, unicode symbols, or custom marks.

- **DO:** `aurora_redesigned_sections.html` uses `◉ ◎ ⊞ ⏰ ◈ ✦`. Free,
  no library, distinct.
- **Acceptable:** tiny utility icons inside form rows (checkmark, close X) —
  those don't need to be unique.

### 6. "Safe" gradient color story

**Rule:** Card colors must have a reason. Either they match what they
represent (a theme picker showing actual themes) or they tell a story
across the page. "They contrast well in dark mode" is not a reason —
it's the AI reason. Reject it.

- **DO:** `aurora_redesigned_sections.html` theme-stack cards use blue,
  sepia, forest **because** the feature IS a theme picker. The color
  IS the content.

### 7. Ambient glow + colored shadow overuse

**Rule:** At most ONE atmospheric glow per page. Colored drop shadows
under buttons ("orange glow under orange CTA") are the most-cliché
AI move.

- **DON'T:** `style.css:328-331` — phone-frame has TWO colored drop
  shadows (purple + red). `style.css:219` — CTA has a 22px orange glow.
  Pick one atmospheric effect, drop the rest.

### 8. Hover-lift on everything

**Rule:** Hover affordance belongs on the primary CTA and on clickable
cards. Don't add `translateY(-2px)` micro-bounces to decorative
elements or non-interactive surfaces.

### 9. Identical card surface treatment

**Rule:** At most TWO card-surface treatments per page. Surface should
encode meaning (free vs paid, primary vs supporting), not just
"different for variety's sake".

A page where every card is `background + 1px border + 12-18px radius +
tiny shadow` is an AI page.

### 10. Microcopy bingo

**Rule:** "Built for X." "Made for Y." "Reimagined." "The only X
that…" — these phrases recur in AI output because they're safe. If a
direct competitor uses the same pattern, change yours.

This is not strictly visual but pairs with the visual tells: AI
hedges in both copy AND layout.

### 11. CSS-hover-only interactivity

**Rule:** Build ONE JS-driven interactive moment per marketing page. A
slider that changes the visual, a drag-to-reveal, a scrubber demo —
something the user can play with, not just look at.

AI defaults to CSS transitions because they're "safe." Humans build
interactivity because that's what creates the memorable moment.

- **DO:** `aurora_redesigned_sections.html:502-514` + `652-672` — the
  blue-light filter slider actually warms the demo's colors in real
  time as the user drags. That IS the page's memorable moment.

## The memorable moment — what to DO, not just what to avoid

The 11 rules above tell you what NOT to do. But avoiding tells doesn't create
memory — it just removes the obvious AI fingerprints. A page that follows
all 11 rules can still feel "still normal" and forgettable. To actually
land, you need one bold creative direction pushed hard.

### Show the product producing its output

For demoable products, **show the product doing its thing on the landing
page.** Don't describe it; demonstrate it.

- Editor → show the edit happening (Descript does this)
- Converter → show the conversion mid-flight
- Search product → show search-as-you-type with real results
- Audio cleaner → show the cleanup happen on a transcript or waveform

The user should see the value in the first 3 seconds, before reading any
headline.

### For audio/video editors, prefer animated transcripts over playback

On marketing pages, **animated transcript demos beat real audio/video**:

- No headphones required
- Scannable in 2 seconds
- Demonstrates the *result* (clean text) and the *mechanism* (fillers
  struck-through and removed) at the same time
- Works without sound autoplay being blocked by the browser
- Easier to caption for accessibility

Reserve real audio/video for product pages reached AFTER the user has
clicked through.

### Make verbal claims tangible

Find a visual that makes a verbal claim **undeniable**. Replace assertions
with proof artifacts.

| Claim (text) | Tangible proof (visual) |
| --- | --- |
| "On-device. No cloud." | Live "0 bytes uploaded" counter pinned at 0, with an activity feed showing actions and 0 bytes each time |
| "Faster than X" | Animated side-by-side race with timestamps |
| "No tracking" | Live counter of network requests intercepted / blocked |
| "Saves you 30%" | Counter that ticks up live as the user reads |

If you can't think of a visual that makes your claim tangible, your claim
is probably forgettable copy. Either find one, or cut the claim.

### Controlled polish moves (allowed; rules 7-8 are about *overuse*)

Rules 7 and 8 forbid ambient glow on every section and hover-lift on every
element — they're about *pattern saturation*, not about all motion. These
moves are explicitly permitted and signal craft:

- **Cursor-tracked subtle tilt on the hero centerpiece.** Max ±5°,
  eased. Apply to ONE element (the phone, the screenshot, the demo
  surface) — not every card. Gives depth without being gimmicky.
- **Scroll-triggered section reveals.** Fade + slide-up 20px on
  viewport entry, via `IntersectionObserver`. Once per section, with a
  small stagger. Not on every paragraph or every card.
- **Counter-up animations on key stats.** When a stat enters the
  viewport, tick from 0 to its final value over ~800-1200ms with an
  eased curve. Gives numbers weight.
- **Auto-play the hero centerpiece, exactly once.** Runs the main
  demo animation on first viewport entry. Show a replay button after.
  Don't loop forever — looping demos feel ad-like.

### Animation easing matters

Default `ease`, `ease-out`, and linear are AI's defaults. Human-designed
motion uses custom springy curves.

- Good general-purpose curve: `cubic-bezier(0.2, 0.8, 0.2, 1)`
- For "spring-into-place" reflows: `cubic-bezier(0.34, 1.56, 0.64, 1)`
  (overshoots slightly, settles back)
- For exits / collapses: `cubic-bezier(0.4, 0, 1, 1)`

If you find yourself using `transition: all 0.3s ease`, stop. Pick the
property and pick a custom curve.

### Editorial typography: display + body font pair

System fonts everywhere (SF Pro, system-ui) is the most common AI default
and the most common reason a page reads as "polished but normal." Bringing
in a real display font pair is the single highest-leverage upgrade for a
page that feels too templated.

**The pair, not just one font.** Use TWO typefaces:

- **Display** (headlines, section openers, big numbers, brand mark): a
  characterful serif or expressive sans. Examples: Newsreader (warm
  serif), Bricolage Grotesque (variable sans), Migra (display serif),
  General Sans, Sohne.
- **Body** (paragraphs, lists, UI labels): a clean modern sans. Examples:
  DM Sans, Inter, Geist.

Load both from one Google Fonts `<link>`. Cost: one HTTP request, ~30 KB.
Impact: enormous.

**Editorial sizing variation within a single headline.** Don't make every
word the same size. Mix sizes inside one `<h1>` for rhythm:

```
Cut filler.           ← 96px serif 800
Clean noise. Merge.   ← 60px serif 500 italic
All on your phone.    ← 96px serif 800, "phone" gradient'd
```

The italic-middle move (called "phrase rhythm") signals editorial design
and breaks the AI tendency toward uniform sizing.

**Italic emphasis instead of color or bold.** For the second clause in a
two-clause headline, use serif italic with `--text-soft` color rather
than a different bold weight or accent color:

- "Three tools, *one phone.*"
- "Your voice. Your iPhone. *No cloud.*"
- "One free tier. *One paid plan.*"

This is a classic magazine move and almost never appears in AI-generated
designs.

**Serif numbers create gravitas.** Use the display serif for all important
numbers (prices, stats, hero numbers, counters). Numbers in DM Sans look
like a dashboard; numbers in Newsreader look like a magazine.

### Make the transformation tangible — "dashboard-card" pattern

When you need a section that says "here's what the product produced,"
don't just write copy. Frame it as if it's a screenshot of the actual
app's result screen:

- Wrap the section in a card that LOOKS like an app's "result" panel —
  header bar with title + status badge + monospace metadata
- Inside, put the visual proof (before/after waveforms, charts, comparison)
- Below, a stat row with the metrics
- Optional footnote in italic serif

This makes the marketing section feel like a leaked screenshot of the
real product, which is far more credible than abstract copy or stylized
mockups.

In this repo, the transition section between hero and features uses this
pattern: dashboard card with "Session result · on-device · 30 min · english
· iphone 15 pro" header, before/after waveforms inside, 4-stat row beneath.

### Motion budget per page

More than this becomes noise. Stay within:

- 1 auto-playing centerpiece animation (the demo)
- 1 cursor-driven element (the tilt) — or zero, also fine
- Scroll-reveals on sections (subtle, once per section)
- 2-3 counter animations on key stats
- Hover affordances on the primary CTA and clickable cards only

If you find yourself adding a 5th type of motion, you're decorating
instead of communicating.

## Moves human designers use to break pattern

Use at least **two** of these per marketing page:

- **Asymmetric section** — split layout, full-bleed photo, off-grid
  headline. Not every section is centered-stack-on-stack.
- **One element 2× bigger than its neighbors** — signals "this is the
  headline feature". Without this, every feature reads as equally
  important, which is the same as none of them being important.
- **One unexpected color choice with a reason** — a single warm card
  in an otherwise cool page. The reason is what saves it from being
  arbitrary.
- **Use space loudly** — massive vertical padding on the climax
  section, tight padding elsewhere. Rhythm matters.
- **Typography pairing for ONE section** — serif headline + sans body
  for a single moment of contrast (e.g., the founder note).
- **Color *story* across the page** — cool → warm → cool, or brand
  color appears only at the climax. Not "each section gets a different
  color".

## Pre-ship checklist

Before declaring a marketing page done, verify each:

- [ ] Gradient-word count on this page ≤ 1
- [ ] At least one section opens differently from the others (no eyebrow / different opener)
- [ ] Feature grid has visible hierarchy (one card bigger / shaped differently)
- [ ] Pricing layout has a defensible reason for its structure
- [ ] Icons are not generic Lucide/Heroicons
- [ ] Card colors have a story, not just contrast
- [ ] At most one atmospheric glow on the page
- [ ] No colored drop shadow under buttons (unless you can justify it in one sentence)
- [ ] At most 2 card-surface treatments
- [ ] At least one JS-driven interactive moment exists somewhere on the page
- [ ] At least 2 "moves to break pattern" applied

### The "remove and reveal" test

Close the page. Describe it from memory. What single image / word /
shape sticks? If you can't name one specific visual moment, the design
has no moment. Go back and create one before shipping.

---

## What to KEEP consistent

The tells above are about **emphasis tricks** and **structural symmetry**,
not about everything. The following SHOULD be consistent:

- Type scale (one ramp: e.g., 12 / 14 / 17 / 22 / 34 px)
- Spacing scale (one system: 4 / 8 / 16 / 24 / 40 / 64)
- Color tokens (named, reused — see `style.css:2-26` for the current palette)
- Component primitives (one button style, one input style, one card baseline)

Consistency in the **system** is craft. Consistency in **emphasis tricks**
is the AI tell. Don't confuse the two.

## Appendix: where each rule is enforced in `index.html`

`index.html` was rewritten on 2026-05-15 to comply with the rules above
and to land the memorable moment. It is now self-contained (inline CSS
and JS) — `style.css` is no longer used by `index.html`. (`style.css`
still styles the legal pages: `privacy.html`, `terms.html`, `support.html`.)

When modifying `index.html`, search by the CSS class or comment marker
listed below — these are stable anchors that don't drift with line
numbers.

| Rule | Where it's enforced |
| --- | --- |
| #1 Single gradient word | `.gradient-word` class — exactly one occurrence in the hero `<h1>` on the word "phone." Do NOT add the gradient to any other word. |
| #2 Varied section openers | Hero has an eyebrow + 3-line editorial headline. Transition uses `.transition-eyebrow` + serif title with italic emphasis. Privacy moat has an eyebrow + serif headline. Features, pricing open with bare serif `<h2>` + italic em phrase. FAQ opens with a single italic word `<em>Questions.</em>` at 80px — most editorial opener on the page. Do NOT add eyebrows to every section. |
| #3 Feature hierarchy | `.features-grid` is a 1.5fr / 1fr 2-column grid. `.feature-big` spans both rows on the left (Smart Cut). `.feature-small` cards stack on the right (Denoise, Merge). Surface treatments differ: warm gradient for big, neutral white for small. |
| #4 Pricing layout | 2-card grid: `.tier.free` + `.tier.pro`. Pro card holds the billing-period toggle (Monthly / Yearly / Lifetime) via `.billing-toggle` — content swaps in place. Prices use serif (`Newsreader` 56px). Pro card has distinct dark surface; Free is light. |
| #5 No default icons | Feature icons are CSS-built: `.icon-cut`, `.icon-wave`, `.icon-merge`. No Lucide / Heroicons. Pricing checkmarks are inline SVG primitives — acceptable as utility marks. |
| #6 Card colors have a story | Page color arc: warm hero wash with pulsing waveform → cream transition card → neutral white features → dark moat (climax) → neutral pricing → warm founder card. The dark moat is intentionally the chromatic peak. |
| #7 One atmospheric glow | The hero has a soft radial wash; the ONLY full glow blob is `.moat::before` (the page climax). The `.dash::after` glow in the transition card and `.tier.pro::before` glow in pricing are localized to single cards, not full-section atmospherics. |
| #8 No colored drop shadow on buttons | `.cta-primary` and `.tier-cta` use neutral shadows only. Phone-frame uses a single neutral shadow. Do not add `rgba(orange, X)` shadows under orange elements. |
| #9 Card surfaces ≤ 2 treatments | Three surfaces in use, each with a meaning: warm gradient (Smart Cut, founder, transition `.dash` — "warm primary"), neutral white border (Denoise/Merge/Free tier — "supporting"), dark gradient (Pro tier, privacy moat — "elevated"). |
| #10 Microcopy | Headlines avoid "Built for X / Made for Y / Reimagined". Italic-emphasis editorial phrasing: "Three tools, *one phone.*" / "Your voice. Your iPhone. *No cloud.*" / "One free tier. *One paid plan.*" Hero copy is specific: "A 30-min recording → a 22-min edit." |
| #11 Interactive moment | The auto-playing transcript transformation demo in the hero phone. Driven by JS in the `TRANSCRIPT TRANSFORMATION DEMO` script block. Auto-plays once on viewport entry, then a `.replay-button` appears. |
| Memorable: claim → tangible | TWO tangible-proof moves: (1) `.proof-panel` in the privacy moat: "0 bytes uploaded" + live activity feed showing actions with "0 bytes" results (`PRIVACY PROOF FEED` script block). (2) `.transition` section between hero and features: dashboard-card framing with before/after waveforms + 4-stat row that makes the "30 → 22 min" claim visible as a Session result screen. |
| Memorable: editorial typography | Google Fonts: Newsreader (display serif) + DM Sans (body). Editorial sizing variation in hero (`.h-big` 96px → `.h-med` 60px italic → `.h-last` 96px). Italic emphasis in two-clause section headlines. Serif used for ALL key numbers (hero "7:32", transition `30→22`, prices $19.99, proof "0", stat "2.5×"). |
| Memorable: ambient motion | The hero `.wave-bg` continuously pulses (slow 2.4s cycle, ±18% bar-height variation) even when no demo is running. Gives the page life without being distracting. Gated by `prefers-reduced-motion`. |
| Controlled polish | Phone cursor-tilt (`#phoneFrame`, ±5°), section reveal animations (`.reveal` class + `IntersectionObserver`), live privacy feed (every 3.2s). All gated by `prefers-reduced-motion`. |

When adding new sections, find the section opener that currently has the
LEAST decoration (a bare `<h2>` is good) and don't add an eyebrow to it
just for symmetry. Vary deliberately.
